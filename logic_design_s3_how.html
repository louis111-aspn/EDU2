How — 순차(Sequential) vs. 일괄(Set-based) 처리

같은 데이터를 다루더라도 행 하나씩 순서대로 처리할 것인가, 조건을 속성으로 보고 집합 단위로 한 번에 처리할 것인가는 코드 복잡도와 성능을 수십 배 바꾼다. 언제 어떤 방식을 선택해야 하는지, 그 판단 기준을 실제 사례로 훈련한다.

📍 핵심 질문 이 조건들, 하나씩 체크해야 하나 한 번에 처리할 수 있나?
⏱ 예상 학습 시간 35–40분
🔗 선수 모듈 S2. Where
두 가지 사고방식: 순서 vs. 속성
순차 처리는 "행을 하나씩 본다". 집합 처리는 "조건을 속성으로 붙인다".
Sequential 순차 처리

데이터를 한 행씩 꺼내서 조건을 확인하고, 조건에 따라 다른 처리를 수행한다. 처리 결과가 다음 행에 영향을 줄 수 있고, 순서가 중요하다. 루프와 IF 분기가 중심이 된다.

사고방식: "이 행은 VIP인가? 그러면 20% 할인. 아니면 GOLD인가? 그러면 15% 할인…"

불가피한 경우
앞 행의 처리 결과가 다음 행의 입력이 될 때. 외부 API 호출 등 행마다 사이드이펙트가 다를 때. 순서 자체가 비즈니스 의미를 가질 때.
Set-based 집합(Set-based) 처리

조건을 "필터"가 아닌 "속성(attribute)"으로 본다. 모든 행에 동시에 조건을 적용해 결과를 한 번에 계산한다. 루프가 없고, 조건이 데이터에 붙는 형태가 된다.

사고방식: "각 행의 등급 속성에 따라 할인율 속성을 동시에 부여한다."

적합한 경우
각 행의 처리가 다른 행과 독립적일 때. 조건이 분류(classification)나 변환(transformation)일 때. 대량 데이터를 DB나 벡터 연산으로 처리할 때.
같은 데이터, 두 가지 처리 방식 비교
❌ 순차 처리: 행을 하나씩 꺼내서 분기
ROW 1 VIP → IF VIP: 20% 0.20
ROW 2 GOLD → ELIF GOLD: 15% 0.15
ROW 3 NEW → ELIF NEW: 5% 0.05
ROW 4 이탈 → ELSE: 0% 0.00
4번의 분기 평가 × 행 수만큼 반복.
등급이 추가될 때마다 분기 수정.
✅ Set-based: 속성을 한 번에 부여
ROW 1 VIP → CASE 동시 적용 0.20
ROW 2 GOLD → CASE 동시 적용 0.15
ROW 3 NEW → CASE 동시 적용 0.05
ROW 4 이탈 → CASE 동시 적용 0.00
전체 행에 CASE WHEN 한 번 실행.
등급 추가 = CASE에 한 줄 추가.
관점의 전환
순차 처리에서는 "이 행이 어떤 조건에 해당하는가"를 묻는다. Set-based에서는 "이 속성값에 어떤 결과가 매핑되는가"를 묻는다. 질문이 바뀌면 코드 구조 전체가 바뀐다.
비교 사례
세 가지 상황에서 순차와 집합 처리를 직접 비교한다
사례 ① 다중 조건 분류
사례 ② 배부 로직
사례 ③ 상태 전이
사례 ① 다중 조건 분류 — "각 주문의 위험도 등급을 산정해"
금액·연체일·신용점수 세 가지 조건의 조합으로 주문마다 위험도를 HIGH / MEDIUM / LOW 로 분류한다. 조건 조합이 많을수록 IF 체인이 폭발적으로 늘어난다.
❌ IF-ELSE 체인 순차 처리
Python — 분기 폭발
def classify_risk(order):
    amount   = order['amount']
    overdue  = order['overdue_days']
    score    = order['credit_score']

    if amount > 10_000_000:
        if overdue > 30:
            return 'HIGH'
        elif score < 600:
            return 'HIGH'
        else:
            return 'MEDIUM'
    elif amount > 1_000_000:
        if overdue > 30:
            return 'MEDIUM'
        elif score < 600:
            return 'MEDIUM'
        else:
            return 'LOW'
    else:
        return 'LOW'

# 조건이 하나 더 추가되면?
# 분기 수가 배로 늘어난다
문제점
조건 3개의 중첩 IF가 이미 8개 분기. 조건이 4개가 되면 16개, 5개면 32개. 분류 기준이 바뀌면 어디를 수정해야 할지 파악조차 어렵다.
✅ 규칙 테이블 + Set-based 매핑
Python — 규칙과 로직 분리
# 규칙을 데이터로 선언
# (우선순위 순서로 나열)
RISK_RULES = [
    {
      'amount_min': 10_000_000,
      'overdue_min': 30,
      'level': 'HIGH'
    },
    {
      'amount_min': 10_000_000,
      'score_max': 600,
      'level': 'HIGH'
    },
    # ... 규칙을 데이터로 추가
]

def classify_risk(order):
    for rule in RISK_RULES:
        if matches(order, rule):
            return rule['level']
    return 'LOW'

# 규칙 추가 = RISK_RULES에
# 딕셔너리 한 줄 추가
개선 포인트
분류 규칙은 데이터로, 분류 로직(matches 함수)은 코드로 분리. 규칙이 바뀌어도 로직은 건드리지 않는다. SQL에서는 CASE WHEN으로 동일 패턴을 구현할 수 있다.
SQL — CASE WHEN으로 Set-based 일괄 분류
-- 전체 주문에 대해 위험도를 한 번에 부여
-- 행 하나씩 Python으로 분기할 필요 없음
SELECT
    order_id,
    amount,
    CASE
        WHEN amount > 10000000
             AND (overdue_days > 30
                  OR credit_score < 600)
            THEN 'HIGH'
        WHEN amount > 1000000
             AND (overdue_days > 30
                  OR credit_score < 600)
            THEN 'MEDIUM'
        ELSE 'LOW'
    END AS risk_level
FROM orders;
-- 100만 행이어도 DB가 한 번에 처리
핵심 전환 포인트
IF-ELSE 체인은 "이 행은 어떤 경우인가"를 물으며 분기한다. CASE WHEN은 "이 속성값에 어떤 결과를 매핑할 것인가"를 선언한다. 질문의 방향이 바뀌면 코드 구조도 바뀐다.
사례 ② 배부 로직 — "공통비를 각 부서에 배분해"
본사 공통비를 각 부서의 인원 수 비율로 배부하는 요구사항. 순차 처리 방식은 겉보기에 자연스럽지만 순서 의존성이라는 구조적 함정이 숨어 있다.
❌ 순차 배부의 순서 의존성 문제
1 영업부 배부 → 잔액 계산
↓ 잔액을 다음 단계 입력으로 사용
2 개발부 배부 → 잔액 계산
↓ 잔액을 다음 단계 입력으로 사용
3 지원부 배부 → 잔액 소진
문제: 배부 순서가 바뀌면 각 부서의 배부액이 달라진다. 스케줄러 순서를 누군가 바꾸거나, 새 부서가 중간에 삽입되면 결과가 틀어진다. 이 순서 의존성은 코드 어디에도 명시되지 않는다.
✅ Set-based 일괄 배부: 순서 없음
① 전체 인원 합계 계산 (1회)
↓ 모든 부서에 동시 적용
— 영업부: 공통비 × (영업 인원 / 전체)
— 개발부: 공통비 × (개발 인원 / 전체)
— 지원부: 공통비 × (지원 인원 / 전체)
개선: 각 부서 배부액은 서로 독립적으로 계산된다. 순서가 없으므로 순서를 바꿔도 결과가 동일하다. 새 부서 추가 = 행 추가.
❌ 순차 배부 (잔액 방식)
Python — 순서 의존 방식
remaining = total_overhead  # 1,000만

# 순서에 따라 결과가 달라짐
for dept in sorted_depts:
    ratio = dept.headcount / total_hc
    allocated = remaining * ratio
    dept.cost = allocated
    remaining -= allocated  # 잔액 갱신

# sorted_depts 순서가 바뀌면
# 각 dept.cost 값이 달라짐
# 마지막 부서가 부동소수점
# 오차를 모두 흡수함
✅ Set-based 일괄 배부
SQL — 순서 독립 방식
-- 전체 인원 합계를 한 번 계산하고
-- 모든 부서에 동시 적용
WITH total AS (
    SELECT SUM(headcount) AS total_hc
    FROM departments
)
SELECT
    d.dept_name,
    d.headcount,
    10000000
      * d.headcount
      / t.total_hc
      AS allocated_cost
FROM departments d
CROSS JOIN total t;

-- 부서 순서와 무관하게
-- 항상 동일한 결과
순차 배부가 불가피한 경우
"1차 배부 결과를 2차 배부의 기준으로 사용해야 한다"처럼 진짜로 앞 단계 결과가 다음 단계 입력이 되어야 할 때는 순차 처리가 맞다. 문제는 그럴 필요가 없는데 습관적으로 순차 처리를 선택하는 것이다. 먼저 "각 행이 서로 독립적인가?"를 질문하라. 독립적이라면 Set-based가 항상 더 낫다.
사례 ③ 상태 전이 — "주문 상태를 단계별로 변경해"
주문이 접수 → 결제완료 → 배송중 → 완료 / 취소 상태를 거친다. 단계별 IF 체크 방식은 코드가 커질수록 "어떤 상태에서 어떤 상태로 갈 수 있나"를 파악하기 어렵게 만든다. 상태 머신(State Machine)은 이 관계를 데이터로 명시한다.
상태 전이 다이어그램
접수
PLACED
결제 완료
▶
결제완료
PAID
출고 처리
▶
배송중
SHIPPING
배송 완료
▶
완료
DONE
PLACED / PAID / SHIPPING 에서 → CANCELLED 전이 가능  |  DONE / CANCELLED 은 최종 상태 (전이 불가)
❌ 단계별 IF 체크
Python — 상태 전이 규칙이 숨어 있음
def cancel_order(order):
    if order.status == 'PLACED':
        order.status = 'CANCELLED'
    elif order.status == 'PAID':
        order.status = 'CANCELLED'
    elif order.status == 'SHIPPING':
        order.status = 'CANCELLED'
    else:
        raise ValueError(
          "취소 불가 상태"
        )

def ship_order(order):
    if order.status == 'PAID':
        order.status = 'SHIPPING'
    else:
        raise ValueError("출고 불가")

# 전이 규칙이 각 함수 안에
# 분산되어 있어 전체 그림이 안 보임
# 새 상태 추가 시 모든 함수 수정
문제점
"어떤 상태에서 어떤 상태로 갈 수 있는가"라는 규칙이 여러 함수에 분산된다. 전이 규칙을 한눈에 볼 방법이 없고, 새 상태가 추가될 때 모든 함수를 수정해야 한다.
✅ 상태 머신: 전이 규칙을 데이터로 선언
Python — 전이 테이블로 명시
# 허용된 전이를 한 곳에 선언
TRANSITIONS = {
  'PLACED':   ['PAID', 'CANCELLED'],
  'PAID':     ['SHIPPING', 'CANCELLED'],
  'SHIPPING': ['DONE', 'CANCELLED'],
  'DONE':     [],  # 최종 상태
  'CANCELLED':[],  # 최종 상태
}

def transition(order, to_status):
    allowed = TRANSITIONS.get(
        order.status, []
    )
    if to_status not in allowed:
        raise InvalidTransitionError(
            f"{order.status} → {to_status}"
        )
    order.status = to_status

# 새 상태? → TRANSITIONS에 한 줄
# 어떤 전이가 허용되나? → 표 한 번에 파악
개선 포인트
전이 규칙(TRANSITIONS)과 전이 로직(transition 함수)이 분리됐다. 전체 상태 다이어그램이 코드에 그대로 표현된다. 이것이 S1에서 소개한 Lookup Table 치환 패턴의 상태 전이 버전이다.
상태 머신이 특히 빛나는 상황
주문, 결재, 티켓, 계약처럼 명확히 정의된 상태와 이벤트가 존재하고, 잘못된 전이를 막아야 하는 도메인에 상태 머신이 적합하다. "DONE 상태에서 취소가 왜 됩니까?"라는 버그 리포트는 대부분 전이 규칙이 코드에 암묵적으로 흩어져 있을 때 발생한다.
순차 처리가 진짜 필요한 경우
Set-based가 항상 우월한 것은 아니다 — 순서가 의미를 가질 때를 구별하라
Set-based를 강조했지만, 순차 처리가 맞는 경우도 분명히 있다. 두 방식을 맹목적으로 선호하는 것이 아니라 왜 순차가 필요한지를 의식적으로 확인하는 것이 핵심이다.
상황 순차가 불가피한 이유 예시
앞 결과가 뒤 입력이 될 때 단계 N의 출력이 단계 N+1의 입력으로 사용되어 병렬화가 불가능함 복리 계산, 재귀적 BOM 원가 롤업, 체인 배부(2차 배부가 1차 결과 기반)
외부 시스템 호출이 행마다 다를 때 API 호출·파일 생성 등 사이드이펙트가 행마다 달라 일괄 처리 불가 고객별 맞춤 이메일 발송, 주문별 배송사 API 호출, 행별 파일 저장
순서 자체가 비즈니스 의미를 가질 때 선착순, 우선순위 큐, 처리 순서에 따라 결과가 달라지는 게 의도된 설계 선착순 쿠폰 발급 (한도 도달 시 중단), 우선순위 기반 자원 배정
이전 상태를 누적해야 할 때 누적 합계, 이동 평균처럼 이전 행의 값이 계산에 포함됨 일별 잔고 계산 (전일 잔고 + 오늘 입출금), 이동 평균, 런닝 토탈
자가 질문 — 순차 처리 선택 전 확인
"이 처리, 순서를 바꿔도 결과가 같은가?"
같다면 → Set-based로 바꿀 수 있다.
다르다면 → 그 순서 의존성이 의도된 비즈니스 요구사항인지, 아니면 설계의 부산물인지를 구별하라. 의도된 것이라면 순차 처리가 맞다. 부산물이라면 Set-based로 리팩터링해야 한다.
How 결정 요약
처리 방식을 선택할 때의 판단 흐름
Q1
각 행의 처리가 다른 행의 결과와 독립적인가?
YES → Set-based 처리 (CASE WHEN, JOIN, 벡터 연산)
NO → Q2로
Q2
순서 의존성이 의도된 비즈니스 요구인가, 설계의 부산물인가?
의도된 것 → 순차 처리, 단 의존 관계를 코드 구조로 명시
부산물 → Set-based로 리팩터링
Q3
조건 분기가 여러 곳에 중복되어 있는가?
YES → 규칙을 데이터(테이블·딕셔너리)로 분리, 로직은 단일 함수로
NO → 현재 구조 유지
Q4
상태와 전이가 복잡하게 얽혀 있는가?
YES → 상태 머신으로 전이 규칙을 선언적으로 명시
NO → 단순 조건 처리로 충분
S3 핵심 요약
순차 처리의 본질적 문제는 속도가 아니라 규칙이 코드 속에 숨는다는 것이다. Set-based 처리는 조건을 데이터(속성)로 끌어내어 규칙을 가시화한다. 어떤 방식을 선택하든, 처리 방식 자체가 설계 의도를 드러내야 한다.
