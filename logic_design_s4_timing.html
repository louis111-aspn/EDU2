Timing — 언제 계산할 것인가

로직 설계에서 간과되기 쉬운 세 번째 축은 타이밍이다. "미리 구워둘 것인가(Pre-compute), 요청이 올 때마다 계산할 것인가(On-demand)"를 결정하는 것은 시스템 전체의 부하 분산과 응답 속도를 좌우한다. 정답은 없고, 트레이드오프를 의식적으로 선택하는 것이 설계다.

계산 비용을 언제 지불할 것인가 — 쓰기 시점과 읽기 시점 사이의 선택
Pre-compute
미리 계산해서 저장해 둔다

데이터가 생성되거나 변경되는 쓰기 시점에 계산을 수행하고 결과를 저장한다. 조회 시점에는 이미 계산된 값을 그대로 읽기만 한다. 빵을 미리 구워서 진열해 두는 방식이다.

비용 지불 시점: 데이터가 바뀔 때 (쓰기)

특성
조회 속도 빠름 · 조회 부하 낮음

결과가 최신이 아닐 수 있음 (허용 가능한 지연)
On-demand
요청이 올 때마다 계산한다

조회 요청이 들어오는 읽기 시점에 실시간으로 계산을 수행한다. 항상 최신 데이터를 기반으로 결과를 만든다. 주문이 들어올 때마다 즉석에서 조리하는 방식이다.

설계 원칙
결정론적(Deterministic) 계산은 Pre-compute, 탐색적(Exploratory) 계산은 On-demand. "같은 입력이면 항상 같은 결과"인 계산은 미리 구워 두는 것이 경제적이다. 반면 사용자가 파라미터를 바꿔가며 탐색하는 계산은 경우의 수가 무한하므로 미리 구워둘 수 없다.

주문이 발생할 때마다 고객의 누적 구매액을 보여주는 기능. 조회가 빈번하고 주문 건수가 수백만 건이 넘어갈 때, 매번 전체 합산 vs. 증분 업데이트 방식이 갈린다.

Pre-compute: 저장된 값 읽기만
❌ 매번 전체 SUM — On-demand
SQL — 조회마다 전체 집계
-- 조회할 때마다 실행됨
-- 주문 1,000만 건이면
-- 매 조회마다 1,000만 행 스캔
SELECT
    customer_id,
    SUM(amount) AS total_purchase
FROM orders
WHERE customer_id = :cid
GROUP BY customer_id;

-- 동시 접속자 100명이면
-- DB가 100개 집계 쿼리를
-- 동시에 실행해야 함
문제점
조회 빈도 × 데이터 볼륨만큼 DB 부하가 선형 증가한다. 주문 이력이 쌓일수록 같은 조회가 점점 느려지는 구조적 문제가 있다.
✅ 증분 업데이트 — Pre-compute
SQL — 주문 시점에 누적액 갱신
-- 주문 저장 시 누적액 즉시 반영
-- (트랜잭션 안에서 함께 실행)
INSERT INTO orders
    (customer_id, amount, ...)
VALUES (:cid, 50000, ...);

INSERT INTO customer_stats
    (customer_id, total_purchase)
VALUES (:cid, 50000)
ON CONFLICT (customer_id)
DO UPDATE SET
    total_purchase =
        customer_stats.total_purchase
        + EXCLUDED.total_purchase;

-- 조회는 단순 SELECT
SELECT total_purchase
FROM customer_stats
WHERE customer_id = :cid;
-- 인덱스 1건 조회, 항상 빠름
개선 포인트
계산 비용을 조회 시점이 아닌 주문 발생 시점으로 이동시킨다. 조회는 항상 인덱스 1건 조회로 일정하게 빠르다. 주문 빈도가 조회 빈도보다 낮다면 Pre-compute가 전체 부하를 낮춘다.
주의 — 증분 업데이트의 함정
Pre-compute 누적 방식은 중간에 데이터를 직접 수정하거나 삭제하면 합계가 틀어진다. 주문 취소, 환불, 데이터 정정이 발생할 때 누적 테이블도 함께 갱신하는 로직이 반드시 필요하다. 이 복잡도가 허용되지 않는다면 On-demand가 더 안전하다.
사례 ② 파생 지표 — 저장 컬럼 vs. 계산 뷰(View)
"영업이익률 = 영업이익 / 매출"처럼 다른 컬럼에서 유도되는 파생 지표. DB에 미리 저장할 것인가, 조회할 때 계산하는 뷰로 둘 것인가. 이 선택은 정확성·유지보수·성능의 트레이드오프다.
⚠ 저장 컬럼 — Pre-compute
저장 파생 컬럼 방식
-- 테이블에 파생 컬럼 저장
ALTER TABLE financials
    ADD COLUMN op_margin DECIMAL(5,2);

-- 데이터 입력/수정 시마다 갱신
UPDATE financials
SET
    op_margin =
        op_income / revenue * 100
WHERE id = :id;

-- 문제: revenue가 0이면?
-- 문제: 산식이 바뀌면
--       전체 행을 UPDATE해야 함
-- 문제: op_income만 바꾸고
--       op_margin 갱신을 빠뜨리면?
위험 요소
산식이 바뀔 때마다 전체 행 UPDATE가 필요하다. 원본 컬럼만 수정하고 파생 컬럼 갱신을 빠뜨리면 데이터가 불일치한다. 파생 지표는 원본에서 결정론적으로 계산되므로 저장이 중복이다.
✅ 계산 뷰 — On-demand
View / Computed Column 방식
-- 파생 지표를 뷰로 정의
-- 저장하지 않고 조회 시 계산
CREATE VIEW v_financials AS
SELECT
    id,
    revenue,
    op_income,
    CASE
        WHEN revenue = 0
            THEN NULL
        ELSE
            op_income::float
            / revenue * 100
    END AS op_margin
FROM financials;

-- 산식이 바뀌면 뷰 하나만 수정
-- 항상 원본 기반으로 계산
-- 불일치 발생 불가능
개선 포인트
파생 지표는 원본에서 항상 유도 가능하므로 저장이 불필요하다. 산식 변경은 뷰 하나만 수정하면 전체에 즉시 반영된다. 원본과 파생값의 불일치가 구조적으로 발생하지 않는다.
파생 지표 저장이 정당화되는 경우
뷰가 아닌 저장 컬럼이 맞는 경우도 있다. 계산 비용이 매우 높고 조회 빈도가 극히 높을 때, 또는 특정 시점의 스냅샷을 영구 보존해야 할 때 (예: 결산 시점의 환율로 환산한 금액). 스냅샷은 "나중에 재계산하면 달라진다"는 점이 저장의 이유다. 단순 산술 파생 지표는 저장하지 않는 것이 원칙이다.
사례 ③ 대시보드 KPI — 실시간 쿼리 vs. 배치 Pre-aggregation
경영진 대시보드에서 "전사 매출 / 주문 건수 / 재고 현황"을 보여주는 요구사항. 사용자 100명이 동시에 접속해서 새로고침을 누를 때, 매번 실시간 집계 vs. 배치로 미리 집계해 두는 방식이 갈린다.

📊

수십~수백 행

🖥️
대시보드 조회
요약 테이블에서
단순 SELECT
❌ 실시간 집계 — On-demand
대시보드 조회 시마다 실행
-- 100명이 동시에 새로고침하면
-- 이 쿼리 100개가 동시 실행됨
SELECT
    DATE_TRUNC('day', order_dt)
        AS day,
    COUNT(*) AS order_cnt,
    SUM(amount) AS revenue,
    SUM(cost) AS total_cost
FROM orders
WHERE order_dt >= '2024-01-01'
GROUP BY 1
ORDER BY 1;
-- 수천만 행 스캔 × 100 동시 실행
-- DB CPU 100% → 전체 서비스 지연
문제점
대시보드 KPI는 1분 전과 지금이 크게 다르지 않은 경우가 대부분이다. 그런데도 매 조회마다 수천만 행을 스캔하면, 동시 접속자가 늘어날수록 DB 전체가 마비된다.
✅ 배치 Pre-aggregation
배치 (매시간 실행)
-- 매시간 배치가 집계 실행
INSERT INTO kpi_daily_summary
    (day, order_cnt, revenue, total_cost)
SELECT
    DATE_TRUNC('day', order_dt),
    COUNT(*),
    SUM(amount),
    SUM(cost)
FROM orders
WHERE order_dt >= NOW() - '1 hour'
GROUP BY 1
ON CONFLICT (day) DO UPDATE ...;

-- 대시보드 조회는 요약 테이블만
SELECT * FROM kpi_daily_summary
ORDER BY day DESC;
-- 수십 행 조회, 0.001초
개선 포인트
집계 비용을 배치 실행 1회로 지불하고, 100명이 동시에 조회해도 부하는 동일하다. "최신에서 최대 1시간 지연"이 비즈니스 요구를 충족한다면 이 방식이 압도적으로 유리하다.
배치 Pre-aggregation이 맞지 않는 경우
실시간성이 진짜로 필요한 경우 — 예를 들어 재고가 0이 되는 순간 즉시 알림이 가야 하거나, 금융 포지션이 실시간으로 위험 한도를 넘는지 모니터링해야 하는 경우 — 배치 지연이 비즈니스 위험을 만든다. 이때는 스트리밍 처리 또는 이벤트 기반 아키텍처가 필요하다. 허용 가능한 지연이 얼마인지를 비즈니스 요구사항으로 명확히 정의하는 것이 먼저다.
캐싱 — 두 전략 사이의 중간 지점
On-demand로 계산하되, 결과를 일정 시간 재사용한다

캐싱은 완전한 Pre-compute(영구 저장)와 순수한 On-demand(매번 계산) 사이의 실용적 중간 전략이다. 처음 요청은 On-demand로 계산하고, 그 결과를 TTL(Time-To-Live) 동안 보관해 재사용한다. TTL이 만료되면 다시 계산해서 갱신한다.

전략 계산 시점 최신성 적합한 상황
Pre-compute
영구 저장		 데이터 변경 시 (쓰기) 변경 즉시 반영 변경 빈도 낮음 + 조회 빈도 높음 + 결정론적 계산
Cache
TTL 보관		 첫 조회 시, TTL 만료 후 재계산 TTL 주기만큼 지연 허용 조회 빈도 높음 + 일정 지연 허용 + 계산 비용 높음
On-demand
매번 계산		 조회 시마다 (읽기) 항상 최신 탐색적 계산 + 경우의 수 무한 + 최신성이 필수
캐시 무효화 — 가장 어려운 문제
캐싱에서 가장 어려운 것은 언제 캐시를 버릴 것인가다. TTL 기반(일정 시간 후 자동 만료)은 단순하지만 지연이 생긴다. 이벤트 기반(원본 데이터 변경 시 즉시 무효화)은 정확하지만 구현이 복잡하다. 캐싱 전략을 도입하기 전에 "이 데이터, 얼마나 오래된 것까지 허용되는가"를 먼저 정의하라.
Timing 결정 플로우
계산 타이밍을 결정할 때의 판단 순서
Q1
이 계산의 결과가 탐색적인가 — 사용자가 파라미터를 바꿔가며 다양한 결과를 볼 것인가?
YES → On-demand. 경우의 수가 무한하므로 미리 구워둘 수 없다.
NO → Q2로
Q2
이 값이 다른 컬럼에서 결정론적으로 유도되는 파생 지표인가?
YES → View / Computed Column. 저장하지 말고 항상 원본에서 계산하라.
NO → Q3로
Q3
조회 빈도가 쓰기(변경) 빈도보다 훨씬 높은가?
YES → Pre-compute 또는 캐싱. 조회 비용을 낮추는 것이 경제적이다.
NO → Q4로
Q4
결과가 최신에서 얼마나 지연되어도 허용되는가?
즉시 → On-demand (또는 이벤트 기반 Pre-compute)
분~시간 허용 → 캐싱 (TTL 설정)
일 단위 허용 → 배치 Pre-aggregation
⚠
Pre-compute 도입 전 반드시 확인할 것
원본 데이터가 수정·삭제될 때 Pre-compute 결과도 함께 갱신하는 로직이 있는가? 없다면 데이터 불일치가 발생한다. 유지 비용이 조회 비용 절감보다 클 수 있다.
S4 핵심 요약
타이밍 선택의 본질은 계산 비용을 언제 지불할 것인가다. Pre-compute는 쓰기 시점에 미리 지불해 조회를 빠르게 만들고, On-demand는 조회 시점에 지불해 항상 최신을 보장한다. 캐싱은 그 사이의 실용적 타협이다. 어떤 전략도 "항상 옳지" 않다 — 허용 지연, 변경 빈도, 조회 빈도 세 가지를 먼저 파악하고 선택하라.
