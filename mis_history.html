핵심 질문

정보시스템은 왜 한 번에 완성되지 않고, 70년에 걸쳐 단계적으로 진화했을까요?

이 자료의 결론

각 단계는 이전 단계의 한계를 보완하며 등장했습니다. SAC(BI+Planning)도 예외가 아니며, 이 진화 흐름의 한 지점일 뿐입니다.

  • 누적적 진화: 새 단계가 등장해도 이전 단계는 사라지지 않고, 그 위에 새로운 레이어가 쌓입니다 (예: ERP는 TPS 기능을 포함하면서 통합을 추가).
  • 3가지 축: 데이터 범위, 의사결정 주체, 시간성이라는 축을 따라가며 각 단계의 위치를 파악합니다.
  • ERP는 전환점: "데이터 통합"이라는 ERP의 가치가 있어야 그 이후의 BI·SAC·AI Agent가 의미를 가집니다.

1950s–1960s TPS
Transaction Processing System · 거래처리시스템

등장 배경

급여 계산, 재고 입출고, 회계 전기처럼 반복적이고 대량인 거래를 사람이 일일이 처리할 수 없게 되면서 등장했습니다.

특징

  • 정형화된 일상 업무 처리 (Operational Level)
  • 거래 단위 기록 (1건의 입고, 1건의 출금)
  • 정확성과 처리 속도가 핵심 가치

한계 → 다음 단계로

데이터는 시스템에 쌓이고 있지만, 경영진은 여전히 감(感)으로 의사결정합니다. "쌓인 데이터를 누가, 어떻게 요약해서 보여줄 것인가?"라는 질문이 남습니다.

1960s–1970s MIS
Management Information System · 경영정보시스템 (정기 보고의 시대)

등장 배경

TPS에 쌓인 거래 데이터를 가공하여, 관리자에게 정기적인 요약 보고서(주간/월간 매출, 재고 현황 등)를 제공하기 시작했습니다.

특징

  • 중간 관리자(Middle Management) 대상
  • 정해진 양식의 정기 보고서(Scheduled Report)
  • "무엇이 일어났는가"를 사후적으로 요약

한계 → 다음 단계로

보고서는 정해진 양식으로만 나옵니다. "이번 달 매출이 왜 떨어졌지? 만약 가격을 10% 내리면 어떻게 될까?" 같은 비정형 질문에는 답할 수 없습니다.

1970s–1980s DSS
Decision Support System · 의사결정지원시스템 (분석의 시대)

등장 배경

MIS의 정형 보고서를 넘어, 관리자가 직접 변수를 바꿔보며 모델 기반 분석을 할 수 있는 도구가 필요해졌습니다.

특징

  • What-if 분석, 시뮬레이션 모델
  • 비정형 / 반정형 의사결정 지원
  • "가격을 10% 내리면 수익은?" 같은 질문에 대응

한계 → 다음 단계로

분석은 가능해졌지만, 각 부서가 자기 데이터로 따로 모델을 만듭니다. 경영진 입장에서는 전사 현황을 한눈에 보기 어렵고, 부서별 분석 결과가 서로 다른 전제를 사용해 일치하지 않습니다.

1980s–1990s EIS
Executive Information System · 경영자정보시스템 (대시보드의 시대)

등장 배경

최고경영진을 위해, 복잡한 분석 결과를 요약된 화면(Dashboard)으로 보여주고, 필요시 세부 내역을 파고들 수 있는(Drill-down) 시스템이 등장했습니다.

특징

  • 경영진(Executive)을 위한 요약 화면
  • Drill-down 개념의 시작
  • 그래픽 중심의 핵심 지표(KPI) 표시

한계 → 다음 단계로

화면은 멋지지만, 그 화면을 채우는 데이터 자체가 부서마다 따로 관리되고 있습니다. 영업 부서의 "매출"과 회계 부서의 "매출"이 일치하지 않는 문제 — 즉 데이터 신뢰성과 일관성의 문제가 근본 원인으로 드러납니다.

1990s–2000s ERP ★ 핵심 전환점
Enterprise Resource Planning · 전사적 자원관리 (통합의 시대)

등장 배경

"EIS가 예쁜 화면을 만들어도, 그 뒤의 데이터가 제각각이면 의미가 없다." — 이 문제의식에서, 영업·생산·재무·인사를 하나의 시스템, 하나의 데이터베이스로 통합하는 ERP가 등장했습니다.

핵심 가치

  • Single Source of Truth: 전사 데이터를 하나의 DB로 통합
  • 모듈 간 실시간 연계 (판매 → 재고 → 생산 → 회계)
  • SAP가 대표 사례 — 이후 모든 분석(BI, SAC)의 데이터 기반

한계 → 다음 단계로

데이터는 통합됐지만, ERP 자체는 "기록하고 처리하는" 시스템입니다. 이 통합된 데이터를 가지고 자유롭게 분석하고, 미래를 계획(Planning)하려면 별도의 도구가 필요합니다.

왜 이 단계가 중요한가

ERP의 "통합된 데이터"가 없었다면, BI나 SAC가 분석할 신뢰할 수 있는 데이터 자체가 존재하지 않았을 것입니다. ERP는 이후 모든 단계의 전제 조건입니다.

2000s–2010s BI
Business Intelligence · 비즈니스 인텔리전스 (시각화와 셀프서비스의 시대)

등장 배경

ERP에 통합된 데이터를 기반으로, IT부서를 거치지 않고 현업 사용자가 직접 리포트와 대시보드를 만들고, 계획(Planning)까지 수행할 수 있는 도구가 발전했습니다.

특징

  • ERP 데이터 기반 리포팅 / 대시보드 / 계획
  • 셀프서비스 분석 — IT 의존도 감소
  • SAC(SAP Analytics Cloud)의 직접적 뿌리

SAC와의 연결

SAC는 전통적 BI(분석·시각화)와 Planning(계획) 기능을 하나의 클라우드 플랫폼으로 통합한 차세대 도구입니다. ERP가 만든 통합 데이터를, BI 단계의 셀프서비스 철학으로 다루는 것이 SAC의 본질입니다.

2020s– AI Agent
자동화와 능동적 의사결정의 시대

등장 배경

BI는 "사람이 질문하면 시스템이 답을 주는" 단계였습니다. AI Agent는 한 단계 더 나아가, 사람이 묻기 전에 스스로 이상치를 탐지하고 원인과 대응안을 제안합니다.

특징

  • 이상치 탐지 및 자동 알림 (Predictive)
  • 자연어 기반 질의 → 자동 분석 (Smart Insights)
  • SAC-P(Planning)의 예측 기능과 연결

SAC와의 연결

SAC의 Smart Insights, Predictive Planning 등 AI 기능은 이 흐름의 일부입니다. 다만 핵심 데이터 기반은 여전히 ERP의 통합 데이터(ACDOCA 등)이며, AI Agent는 그 위에 올라가는 마지막 레이어입니다.

SUMMARY

진화의 패턴 정리

7단계의 변화를 3가지 축(MECE)으로 정리하면, SAC의 위치가 명확해집니다.

단계 ① 데이터 범위 ② 의사결정 주체 ③ 시간성
TPS개별 거래시스템이 처리현재 (실시간 기록)
MIS부서 단위시스템이 요약 → 사람이 확인과거 (정기 보고)
DSS부서 단위 모델사람이 분석과거 + 가상의 미래(시뮬레이션)
EIS부서 데이터의 요약 화면경영진이 확인과거 요약
ERP전사 통합시스템이 처리 + 통합현재 (실시간 통합 기록)
BI / SAC전사 통합 데이터사람이 분석 + 계획과거 분석 + 미래 계획
AI Agent전사 + 산업 데이터AI가 보조 / 제안과거 + 현재 + 미래 예측

핵심 결론

SAC는 세 축이 모두 교차하는 지점에 위치합니다 — ① ERP가 통합한 전사 데이터를 기반으로, ② 사람이 직접 분석·계획하면서, ③ 과거 분석과 미래 계획을 한 플랫폼에서 수행합니다. 이것이 SAC가 단순한 "리포팅 툴"이 아니라 ERP 이후 진화의 자연스러운 다음 단계인 이유입니다.

상황 A

"매월 말 각 부서가 엑셀로 실적을 정리해서 메일로 보냅니다. 분석은 각자 알아서 합니다."

상황 B

"경영진용 대시보드는 있지만, 부서별 숫자가 서로 안 맞아서 결국 회의에서 다시 맞춰봐야 합니다."

상황 C

"ERP로 데이터는 통합되어 있지만, 분석이나 계획(Planning)은 여전히 엑셀로 다운로드해서 작업합니다."

상황 D

"ERP 데이터를 기반으로 셀프서비스 대시보드와 계획 시나리오를 직접 만들고 있습니다."
