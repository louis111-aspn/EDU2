Where — 처리 레이어를 어디에 둘 것인가

같은 로직도 어느 레이어에서 실행하느냐에 따라 성능, 보안, 재사용성이 완전히 달라진다. DB인지, 앱인지, 프론트엔드인지를 결정하는 것은 취향이 아니라 판단 기준에 따른 설계 결정이다.

📍 핵심 질문 이 로직, 어느 레이어에 두는 게 맞나?

레이어
강점 · 적합한 역할
약점 · 피해야 할 역할
DB
Database Layer
✅ 여기서 잘하는 것 대량 데이터 집계 · 필터 · 조인을 데이터 이동 없이 처리. 트랜잭션 무결성 보장. 인덱스 기반 고속 조회. 감사(Audit) 추적이 필요한 데이터 변경 이력 관리. 여러 앱이 공통으로 참조하는 규칙과 마스터 데이터 보관.
❌ 여기서 하면 안 되는 것 이메일 발송·API 호출 등 외부 시스템 연동. 사용자별로 달라지는 표현 로직 (UI 포맷, 언어 변환). 자주 바뀌는 비즈니스 규칙 (DB 스키마 변경 비용이 높음). 탐색적·실험적 계산 (쿼리 변경 사이클이 느림).
앱
Application / Middleware
✅ 여기서 잘하는 것 비즈니스 규칙 오케스트레이션 (조건 분기, 흐름 제어). 여러 데이터 소스 취합·변환. 외부 시스템 연동 (메일, 결제, 알림). 권한 검증과 인증·인가 처리. 재사용 가능한 서비스 단위 캡슐화.
❌ 여기서 하면 안 되는 것 수백만 행을 메모리에 올려 집계 (메모리·속도 한계). 데이터 무결성 강제 (DB 제약이 더 안전). 사용자 인터랙션에 즉각 반응해야 하는 UI 상태 관리.
프론트
Frontend / Client
✅ 여기서 잘하는 것 사용자 인터랙션에 즉각 반응 (입력값 검증, 실시간 계산). 서버 왕복 없는 로컬 상태 조작 (장바구니, 폼 상태). 표현 로직 (날짜 포맷, 금액 단위, 언어 변환). 작은 데이터셋 필터·정렬 (이미 로드된 목록).
❌ 여기서 하면 안 되는 것 보안 관련 판단 (클라이언트 코드는 조작 가능). 대량 데이터 처리 (브라우저 메모리·성능 한계). 비즈니스 핵심 규칙 단독 보관 (다른 채널 진입 시 우회됨).
핵심 원칙
레이어는 각자의 강점 영역이 있다. 문제는 대부분 한 레이어가 다른 레이어의 역할을 침범할 때 발생한다. 레이어 월경(Layer Crossing)은 단기적으로 편리해 보이지만, 시스템이 커질수록 유지보수 비용이 폭발적으로 증가한다.
판단 기준 4가지
"어디서 처리할까"를 결정할 때 반드시 따져야 하는 체크리스트
01
데이터 볼륨
처리할 데이터가 얼마나 되는가. 수십 행이라면 어디서 처리해도 무방하지만, 수백만 행이라면 데이터를 이동시키지 않고 DB에서 처리하는 것이 압도적으로 유리하다. 네트워크를 타고 앱으로 데이터를 끌어올려 집계하는 것은 DB에서 GROUP BY 하는 것보다 수십 배 느리고 메모리를 과소비한다.
💬 "이 데이터를 앱으로 가져오면 몇 MB인가? GB인가?"
02
재사용성
이 로직을 여러 화면, 여러 앱, 여러 채널이 공통으로 사용하는가. 프론트에서만 구현한 비즈니스 규칙은 모바일 앱, API, 배치 잡이 같은 규칙을 적용받지 못한다. 재사용 빈도가 높을수록 더 낮은 레이어(DB나 앱 서비스)로 내려야 한다.
💬 "이 로직, 웹 말고 다른 곳에서도 써야 하나?"
03
감사 추적 필요 여부
"누가 언제 이 데이터를 바꿨는가"를 추적해야 하는가. 금융 거래, 승인 이력, 재고 변경 등은 감사 로그가 필수다. 프론트나 앱 레이어에서만 처리하면 로그가 남지 않거나 쉽게 우회된다. DB 레이어에서 트리거나 감사 테이블로 처리해야 신뢰할 수 있는 이력이 남는다.
💬 "나중에 '누가 바꿨냐'는 질문이 올 수 있나?"
04
실시간성
사용자가 입력할 때마다 즉각 반응해야 하는가, 아니면 조금 지연되어도 괜찮은가. 실시간 반응이 필요한 UI 피드백(입력값 유효성, 금액 계산)은 프론트에서 처리해야 한다. 서버 왕복(Round Trip)이 생기면 사용자는 지연을 체감한다. 반면 정확성이 중요한 최종 검증은 반드시 서버에서 한 번 더 해야 한다.
💬 "사용자가 이 결과를 얼마나 빨리 봐야 하나?"

사례 ① 대량 집계 — "월별 지역별 매출 합계를 보여줘"
1,000만 건의 주문 데이터에서 월×지역 집계표를 만드는 요구사항. 어디서 처리하느냐가 성능을 수십 배 가른다.
❌ 앱 레이어에서 집계
앱 서버 (Python) — 잘못된 방식
# DB에서 전체 데이터를 끌어와서
# 앱에서 직접 집계한다
rows = db.execute(
    "SELECT * FROM orders"
)  # ← 1,000만 행 전송

result = {}
for row in rows:
    key = (row.month, row.region)
    result[key] = (
        result.get(key, 0)
        + row.amount
    )
# 메모리: 수 GB 점유
# 시간: 수십 초
문제점
1,000만 행을 네트워크로 전송해서 앱 메모리에 올린 뒤 Python 루프로 집계. DB가 인덱스와 병렬 처리로 0.1초에 끝낼 일을 앱이 30초 넘게 하고 있다.
✅ DB 레이어에서 집계
SQL — 올바른 방식
-- 집계를 DB에서 완결한다
-- 네트워크로 이동하는 행 수:
-- 월 수 × 지역 수 = 수십 행
SELECT
    DATE_TRUNC('month', order_dt)
        AS month,
    region,
    SUM(amount) AS total
FROM orders
GROUP BY 1, 2
ORDER BY 1, 2;

-- 결과: 수십 행만 앱으로 전송
-- 시간: 인덱스 활용 시 수백 ms
개선 포인트
데이터는 이동시키지 않고 DB 안에서 집계를 완결한다. 앱으로 넘어오는 것은 결과 수십 행뿐. 데이터는 태어난 곳에서 처리한다는 원칙이 여기서 적용된다.
판단 근거 적용
데이터 볼륨 기준 → DB. 1,000만 행은 앱으로 이동시키는 순간 네트워크·메모리 병목이 발생한다. 집계 자체는 DB가 월등히 잘한다. 앱의 역할은 결과를 받아서 화면에 그리는 것으로 한정한다.
사례 ② 유효성 검증 — "이메일 형식이 맞는지 확인해"
검증 로직은 프론트, 앱, DB 모두에 둘 수 있다. 어디에 두느냐에 따라 사용자 경험과 보안이 달라진다. 핵심은 각 레이어가 다른 목적의 검증을 담당한다는 것이다.
세 레이어에서 검증이 수행하는 서로 다른 역할
레이어 무엇을 검증하나 목적 없으면 생기는 문제
프론트엔드 형식 (이메일 패턴, 필수 항목 빈칸) 사용자에게 즉각 피드백 — UX 서버 왕복 후 에러 반환, 느린 피드백
앱 서버 비즈니스 규칙 (중복 이메일, 도메인 화이트리스트) 보안·규칙 강제 — 최종 게이트 프론트 우회 시 (API 직접 호출) 검증 없음
DB 무결성 제약 (UNIQUE, NOT NULL, FK) 데이터 일관성 최후 보루 앱 버그로 중복 삽입, 고아 레코드 발생
❌ 프론트에서만 검증
JavaScript — 위험한 방식
// 프론트에서만 중복 이메일 체크
function validateEmail(email) {
  const existing = users.find(
    u => u.email === email
  );
  if (existing) {
    alert("이미 사용 중인 이메일");
    return false;
  }
  return true;
}

// 문제: Postman이나 curl로
// API를 직접 호출하면?
// → 프론트 검증 완전 우회 가능
문제점
클라이언트 코드는 누구나 조작·우회할 수 있다. 비즈니스 규칙을 프론트에만 두면 API를 직접 호출하는 모든 경로가 무방비 상태가 된다.
✅ 레이어별 역할 분담
올바른 3-레이어 검증 구조
--- 프론트: UX 피드백 (형식) ---
const isValid = /^[^@]+@[^@]+$/
    .test(email);
// 즉각 피드백, 서버 왕복 없음

--- 앱 서버: 비즈니스 규칙 ---
def register(email):
    if User.exists(email=email):
        raise DuplicateEmailError
    # 어떤 경로로 와도 여기서 막힘

--- DB: 무결성 최후 보루 ---
CREATE TABLE users (
    email VARCHAR(255)
        UNIQUE NOT NULL
    -- 앱 버그도 여기서 차단
);
개선 포인트
세 레이어가 서로 다른 목적으로 검증한다. 프론트는 UX를 위해, 앱은 보안을 위해, DB는 무결성을 위해. 중복이 아니라 각자 다른 위협을 막는 것이다.
흔한 오해
"앱에서도 체크하고 DB에서도 체크하면 중복이 아닌가?" — 아니다. 앱 검증은 비즈니스 규칙(빠른 피드백)을 위한 것이고, DB 제약은 앱 자체에 버그가 있을 때도 데이터를 지키는 최후의 보루다. 두 레이어의 검증은 서로 다른 적을 막고 있다.
사례 ③ 권한 필터링 — "이 사용자가 볼 수 있는 데이터만 보여줘"
Row-level 보안: 부서장은 자기 부서 데이터만, 지역 매니저는 자기 지역 데이터만 볼 수 있어야 한다. 어디서 필터링하느냐가 보안과 성능을 동시에 결정한다.
❌ 앱 레이어에서 필터링
앱 서버 — 취약한 방식
# 전체 데이터를 먼저 가져와서
# 앱에서 권한에 따라 필터
all_orders = db.execute(
    "SELECT * FROM orders"
)

def filter_by_user(user, orders):
    return [
        o for o in orders
        if o.dept_id == user.dept_id
    ]

result = filter_by_user(
    current_user, all_orders
)
# 문제 1: 전체 데이터 전송 후 필터
# 문제 2: 새 API 엔드포인트 추가 시
#         필터 적용을 빠뜨릴 수 있음
문제점
전체 데이터를 앱으로 가져온 뒤 걸러낸다 — 비효율적이다. 더 심각한 문제는 새 API 엔드포인트를 추가할 때 필터 적용을 빠뜨리면 권한 없는 데이터가 노출된다. 보안 로직이 앱 코드 여러 곳에 분산되어 있어 누락이 생기기 쉽다.
✅ DB 레이어에서 필터링
SQL Row-Level Security — 안전한 방식
-- DB에 Row-Level Security 설정
-- (PostgreSQL 예시)
ALTER TABLE orders
    ENABLE ROW LEVEL SECURITY;

CREATE POLICY dept_isolation
    ON orders
    USING (
        dept_id = current_setting(
            'app.current_dept'
        )::INT
    );

-- 이제 어떤 쿼리를 써도
-- 자동으로 부서 필터 적용됨
-- 새 API가 추가되어도 안전
SELECT * FROM orders;
-- → 자동으로 내 부서 것만 반환
개선 포인트
권한 필터를 DB에 내장하면 어떤 쿼리, 어떤 API 경로로 접근해도 항상 적용된다. 앱 개발자가 필터 적용을 "기억"해야 하는 부담이 사라진다. 보안이 인프라 레벨에서 강제된다.
판단 근거 적용
재사용성 + 감사 추적 기준 → DB. 권한 규칙은 웹, API, 배치, 리포트 등 모든 접근 경로에 일관되게 적용되어야 한다. 앱 레이어에 두면 경로가 늘어날 때마다 적용을 확인해야 하는 부담이 생긴다. DB 레이어에 내장하면 한 번 설정으로 모든 경로가 보호된다.
레이어 선택 판단 요약
요구사항을 받았을 때 이 흐름으로 레이어를 결정하라
Where 결정 플로우차트
Q1
사용자 인터랙션에 즉각 반응해야 하나?
YES → 프론트에서 처리 (단, 최종 검증은 서버에서 반드시 한 번 더)
NO → Q2로
Q2
처리할 데이터가 수십만 행 이상인가?
YES → DB에서 처리 (집계, 필터, 조인을 DB에서 완결)
NO → Q3로
Q3
여러 채널(웹/앱/API/배치)에서 공통으로 사용하나?
YES → 앱 서비스 레이어에 캡슐화 (한 곳에서 관리)
NO → Q4로
Q4
감사 추적이나 무결성 보장이 필요한가?
YES → DB 레이어에서 제약·트리거·감사 테이블로 처리
NO → 앱 레이어에서 처리 (비즈니스 로직 오케스트레이션)
⚠
보안 관련 판단은 절대 프론트에만 두지 말 것.
클라이언트 코드는 누구나 수정·우회할 수 있다. 권한 검증, 데이터 접근 제어, 중요 비즈니스 규칙은 반드시 서버(앱 또는 DB) 레이어에서 강제한다.
S2 핵심 요약
레이어 선택에 정답은 없지만 판단 기준은 있다. 볼륨이 크면 DB, 재사용이 필요하면 앱 서비스, 즉각 반응이 필요하면 프론트 — 그리고 보안은 항상 서버에서 한 번 더. 레이어를 넘나드는 설계의 대부분은 이 네 가지 기준 중 하나를 무시했을 때 발생한다.
