로직 설계 감각이란 무엇인가

"돌아가는 코드"와 "잘 설계된 로직"은 다르다. 설계 감각은 특정 언어나 프레임워크의 숙련도가 아니라, 어디서 · 언제 · 어떻게 처리할지를 판단하는 직관이다. 이 섹션은 그 직관을 훈련하기 위한 프레임과, 나쁜 설계가 풍기는 냄새를 다룬다.

📍 대상 구현 경험 보유 컨설턴트
⏱ 예상 학습 시간 25–30분
🔗 선수 지식 없음 (시리즈 첫 모듈)
돌아가는 코드 vs. 잘 설계된 로직
같은 요구사항, 전혀 다른 설계 품질

아래 두 코드는 "주문 상태에 따라 할인율을 반환한다"는 동일한 요구사항을 구현한다. 실행 결과는 동일하지만, 3개월 뒤 요구사항이 바뀌었을 때 어떤 일이 벌어지는지 보자.

❌ 돌아가는 코드
def get_discount(status):
    if status == "VIP":
        return 0.20
    elif status == "GOLD":
        return 0.15
    elif status == "SILVER":
        return 0.10
    elif status == "NEW":
        return 0.05
    else:
        return 0.00

# 6개월 뒤: PLATINUM 등급 추가
# → 함수 내부를 직접 수정해야 함
# → 어디에 몇 개가 있는지 찾아야 함
문제점
새 등급이 생길 때마다 코드를 수정해야 한다. 같은 패턴이 3곳에 있으면 3곳을 모두 찾아야 한다. 비즈니스 규칙이 코드에 하드코딩되어 있다.
✅ 잘 설계된 로직
# 규칙을 데이터(테이블)로 분리
DISCOUNT_RULES = {
    "VIP":    0.20,
    "GOLD":   0.15,
    "SILVER": 0.10,
    "NEW":    0.05,
}

def get_discount(status):
    return DISCOUNT_RULES.get(
        status, 0.00
    )

# 6개월 뒤: PLATINUM 추가
# → 테이블에 한 줄만 추가
# → 함수 로직은 건드리지 않음
개선 포인트
비즈니스 규칙(할인율)과 처리 로직(get_discount)이 분리됐다. 규칙이 바뀌어도 로직은 변하지 않는다. 이것이 Lookup Table 치환 패턴이다 (Section 5에서 심화).
핵심 관점
설계 품질의 진짜 척도는 "지금 돌아가는가"가 아니라 "요구사항이 바뀌었을 때 얼마나 적은 곳을 수정해도 되는가"다. 좋은 설계는 변경의 파급 범위를 최소화한다.
설계 판단의 3축 프레임
모든 로직 설계 결정은 세 가지 질문으로 분해된다
Axis 1
Where?
어느 레이어에서 처리할 것인가
DB인가, 앱(미들웨어)인가, 프론트엔드인가. 같은 계산도 어디서 실행하느냐에 따라 성능, 보안, 재사용성이 완전히 달라진다.
DB 집계 앱 검증 프론트 렌더
Axis 2
When?
언제 계산할 것인가
미리 구워둘 것인가(Pre-compute), 요청이 올 때마다 즉석에서 계산할 것인가(On-demand). 타이밍이 시스템 전체의 부하 분산을 결정한다.
배치 Pre-compute 실시간 On-demand 캐싱
Axis 3
How?
어떤 방식으로 처리할 것인가
행 하나씩 순차 처리(Sequential)할 것인가, 조건을 속성으로 보고 집합 단위로 일괄 처리(Set-based)할 것인가. 처리 방식이 코드 복잡도를 결정한다.
순차 IF 분기 Set-based 파이프라인
요구사항이 들어왔을 때 3축을 순서대로 질문하는 흐름

주의
현실에서는 세 축이 독립적이지 않다. Where를 DB로 결정하면 How 선택지가 Set-based로 자연스럽게 좁혀진다. 세 축을 동시에 고려하는 습관이 설계 감각의 핵심이다.
나쁜 설계의 냄새
Anti-pattern 5가지 — 냄새를 맡는 것부터가 설계 감각의 시작

코드 리뷰에서, 또는 자신이 짠 코드를 한 달 뒤 다시 볼 때 "이거 왜 이렇게 짰지?" 싶은 순간이 있다. 그 순간들에는 패턴이 있다. 아래 다섯 가지는 가장 흔히 나타나는 나쁜 설계의 냄새다. 각 항목을 눌러 냄새의 원인과 처방을 확인하라.

AP-01 중복 분기 (Duplicate Branching) 같은 조건 판단이 여러 곳에 흩어져 있다 +

냄새: "VIP이면 할인" 로직이 주문 모듈, 장바구니 모듈, 알림 모듈에 각각 IF 문으로 존재한다. 어느 날 VIP 기준이 바뀌면 세 군데를 모두 찾아서 고쳐야 한다. 그리고 반드시 하나는 빠뜨린다.

원인: 비즈니스 규칙을 코드에 직접 넣고, 그 코드를 복사해서 씀.

처방: 규칙을 한 곳에 정의하고 나머지는 그것을 참조한다. 함수 하나, 테이블 하나, 또는 설정 파일 하나로 모든 판단의 단일 진실 공급원(Single Source of Truth)을 만든다.

AP-02 순서 의존 (Order Dependency) A를 먼저 실행해야만 B가 올바르게 동작한다 +

냄새: 배치 잡이 특정 순서로 실행되어야 한다. 누군가 스케줄러 순서를 바꾸면 결과가 틀어진다. 새 팀원은 이 순서를 알 방법이 없다. 주석으로만 존재한다.

원인: 중간 상태를 공유 테이블에 저장하고, 다음 단계가 그 상태를 전제로 설계됨. 또는 순차 배부처럼 앞 단계 결과가 뒤 단계의 입력이 되는 구조를 제거하지 않음.

처방: 각 단계를 독립적으로 만든다. 입력이 같으면 항상 같은 출력이 나와야 한다(멱등성). 순서 의존이 불가피하면 의존 관계를 코드 구조(DAG 등)로 명시한다.

AP-03 매직 넘버 / 매직 스트링 (Magic Literals) 의미를 알 수 없는 숫자와 문자열이 코드에 직접 박혀 있다 +

냄새: if amount > 500000 — 왜 50만인가? 누가 정했나? 바뀌면 어디를 고쳐야 하나? status == "A3" — A3가 무슨 의미인지 코드만 보면 알 수 없다.

원인: 비즈니스 의미를 가진 값을 이름 없이 코드에 직접 씀.

처방: 의미 있는 상수명이나 Enum으로 추출한다. BULK_ORDER_THRESHOLD = 500000, Status.APPROVED = "A3". 이름이 문서가 된다.

AP-04 레이어 월경 (Layer Crossing) 프론트엔드가 DB 쿼리를 직접 조립하거나, DB가 UI 렌더링 로직을 갖는다 +

냄새: 화면단 JavaScript에서 여러 테이블을 직접 조인하는 쿼리를 문자열로 만들어 서버에 보낸다. 또는 SQL Stored Procedure 안에 이메일 발송 로직이 들어가 있다.

원인: 빠른 개발을 위해 "일단 동작하게" 만들다 보면 레이어 경계가 무너진다. 각 레이어가 무엇을 책임지는지 명확한 기준이 없을 때 발생한다.

처방: 각 레이어의 책임을 명확히 정의한다. DB는 데이터 저장과 집계, 앱은 비즈니스 로직과 오케스트레이션, 프론트는 표현과 인터랙션. 경계를 넘어야 할 이유가 생기면 그것이 설계를 재검토하라는 신호다.

AP-05 암묵적 상태 (Implicit State) 어딘가에 "전제 조건"이 있는데 코드 어디에도 명시되지 않았다 +

냄새: 함수 A가 전역 변수 current_user를 직접 읽는다. 이 함수를 다른 맥락에서 호출하면 누구의 current_user인지 알 수 없다. 배치 환경에서는 아예 없는 변수다.

원인: 함수가 자신의 입력 이외의 것에 의존한다(Side Effect, Hidden Dependency). 함수 시그니처를 보는 것만으로는 무엇이 필요한지 알 수 없다.

처방: 함수에 필요한 모든 것은 인자로 전달한다. 전역 상태 대신 명시적 파라미터. 함수 시그니처 자체가 계약서가 되어야 한다.

지금 나의 설계 감각은?
코드를 짜기 전, 이 세 가지 질문을 습관처럼 던지는가
# 질문 던지지 않으면 던지는 습관이 생기면
W 이 로직, 어느 레이어에서 처리하는 게 맞나? 프론트가 DB 역할을 하거나, DB가 앱 로직을 갖게 됨 레이어별 책임이 명확해지고 성능 문제가 사전에 예방됨
H 이 조건들, 순서대로 체크해야 하나 아니면 한 번에 처리할 수 있나? 불필요한 루프와 IF 체인이 쌓이고 순서 의존성이 생김 Set-based 접근으로 코드가 단순해지고 성능이 향상됨
T 이 계산, 미리 해두는 게 나은가 매번 하는 게 나은가? 화면이 열릴 때마다 무거운 집계 쿼리가 실행됨 조회 성능과 계산 비용의 균형을 의식적으로 설계함
이 시리즈의 목표
위 세 질문을 코드를 짜기 전에 자동으로 던지는 것이 이 시리즈의 목표다. S2(Where), S3(How), S4(Timing)에서 각 축을 실제 사례와 함께 훈련하고, S5에서 반복적으로 쓸 수 있는 패턴으로 정리한다. S6에서 자신의 판단을 검증하라.
