01
Lookup Table 치환
IF 체인을 테이블 조회로 대체해 규칙과 로직을 분리한다
How — Set-based Timing — Pre-compute 가능
언제 쓰는가
동일한 분기 패턴이 여러 곳에 반복되거나, 분기의 조건이 비즈니스 규칙(요금표·등급표·코드 정의)일 때. 규칙이 자주 바뀌거나, 새 항목이 추가될 가능성이 있을 때. "새 항목이 생기면 코드를 수정해야 한다"는 생각이 드는 순간이 이 패턴의 적용 신호다.
패턴 구조
Input
조건값
"VIP", "GOLD" …
→
Lookup
규칙 테이블
key → value 매핑
→
Output
결과값
0.20, 0.15 …
로직(함수)은 변경 없이 유지  |  규칙(테이블)만 추가·수정
❌ 적용 전 — IF 체인
비즈니스 규칙이 코드에 하드코딩
def get_tax_rate(country):
    if country == "KR":
        return 0.10
    elif country == "US":
        return 0.08
    elif country == "DE":
        return 0.19
    elif country == "JP":
        return 0.10
    elif country == "FR":
        return 0.20
    else:
        return 0.00

# 새 국가 추가 → 코드 수정
# 세율 변경 → 코드 수정
# 같은 패턴이 3군데 있으면?
✅ 적용 후 — Lookup Table
규칙을 데이터로, 로직은 단일 함수로
# 규칙: 데이터로 선언
TAX_RATES = {
    "KR": 0.10,
    "US": 0.08,
    "DE": 0.19,
    "JP": 0.10,
    "FR": 0.20,
}

# 로직: 테이블 조회
def get_tax_rate(country):
    return TAX_RATES.get(
        country, 0.00
    )

# 새 국가: TAX_RATES에 한 줄
# 세율 변경: 값 하나만 수정
# 로직 함수는 영원히 그대로
SQL 버전 — DB 테이블을 Lookup으로 사용
-- 규칙을 DB 테이블로 관리하면
-- 코드 배포 없이 규칙을 변경할 수 있다
CREATE TABLE tax_rules (
    country     VARCHAR(2) PRIMARY KEY,
    tax_rate    DECIMAL(4,2),
    valid_from  DATE,
    valid_to    DATE
);

-- 주문에 세율 적용 — JOIN으로 처리
SELECT
    o.order_id,
    o.amount,
    o.amount * COALESCE(t.tax_rate, 0)
        AS tax_amount
FROM orders o
LEFT JOIN tax_rules t
    ON  o.country   = t.country
    AND o.order_dt BETWEEN
        t.valid_from AND t.valid_to;
-- 새 국가 추가 = INSERT 한 줄
-- 세율 변경 = UPDATE 한 줄, 배포 불필요
쓰면 안 되는 경우
각 분기 안의 로직이 단순한 값 반환이 아니라 복잡한 처리 흐름을 포함할 때. 예를 들어 "VIP면 A 처리 후 B를 호출하고, GOLD면 C만 한다"처럼 동작 자체가 다를 때는 Lookup Table보다 전략 패턴(Strategy Pattern)이 적합하다. Lookup Table은 값의 매핑에 특화된 패턴이다.

02
파이프라인 분해
하나의 복잡한 로직을 독립적인 단계별 변환으로 쪼갠다
How — 구조화 Where — 레이어 분리
언제 쓰는가
하나의 함수나 쿼리가 "데이터 정제 + 변환 + 집계 + 포맷"을 한꺼번에 처리하고 있을 때. 중간 단계를 테스트하거나 디버깅하기 어려울 때. 특정 단계만 교체하거나 순서를 바꿔야 하는 요구가 생겼을 때. "이 함수가 뭘 하는지 한 문장으로 설명할 수 없다"는 생각이 드는 순간이 적용 신호다.
Stage 1
수집 / 로드
Raw Input
↓ 원본 데이터
→
Stage 2
정제 / 검증
원본 데이터
↓ 유효 데이터
→
Stage 3
변환 / 계산
유효 데이터
↓ 변환 결과
→
Stage 4
집계 / 요약
변환 결과
↓ 집계값
→
Stage 5
출력 / 저장
집계값
↓ 최종 결과
❌ 적용 전 — 한 함수에 모든 것
단일 함수가 모든 단계를 처리
def process_sales_report(raw):
    # 정제
    cleaned = []
    for row in raw:
        if not row.get('amount'):
            continue
        if row['amount'] < 0:
            continue
        cleaned.append(row)
    # 변환
    converted = []
    for row in cleaned:
        row['usd'] = (
            row['amount'] * 0.00075
        )
        converted.append(row)
    # 집계
    total = sum(
        r['usd'] for r in converted
    )
    return {'total_usd': total}

# 정제 로직만 테스트하려면?
# 환율만 바꾸려면?
# 불가능 — 함수 전체를 건드려야 함
✅ 적용 후 — 단계별 함수 분리
각 단계가 독립 함수, 파이프라인으로 연결
def clean(rows):
    return [
        r for r in rows
        if r.get('amount', 0) > 0
    ]

def convert_to_usd(rows, rate):
    return [
        {**r, 'usd': r['amount'] * rate}
        for r in rows
    ]

def aggregate(rows):
    return {
        'total_usd': sum(
            r['usd'] for r in rows
        )
    }

# 파이프라인 연결
def process_sales_report(raw):
    return aggregate(
        convert_to_usd(
            clean(raw), rate=0.00075
        )
    )
# clean만 테스트 가능
# 환율 변경: rate 파라미터만
SQL 버전 — CTE로 파이프라인 구현
-- 각 CTE가 하나의 파이프라인 단계
-- 중간 결과를 이름으로 참조 가능 → 디버깅 용이
WITH
-- Stage 1: 정제
cleaned AS (
    SELECT * FROM raw_sales
    WHERE amount > 0
      AND order_dt IS NOT NULL
),
-- Stage 2: 변환 (환율 적용)
converted AS (
    SELECT
        *,
        amount * 0.00075 AS amount_usd
    FROM cleaned
),
-- Stage 3: 집계
aggregated AS (
    SELECT
        DATE_TRUNC('month', order_dt) AS month,
        SUM(amount_usd) AS total_usd
    FROM converted
    GROUP BY 1
)
SELECT * FROM aggregated ORDER BY month;
-- 디버깅: SELECT * FROM cleaned; 로 중간 단계 확인 가능
쓰면 안 되는 경우
단계가 2개 이하이고 로직이 단순할 때 — 과도한 분해는 오히려 코드를 읽기 어렵게 만든다. 각 단계 사이에 상태를 공유해야만 하는 경우 — 파이프라인은 각 단계가 독립적일 때 작동한다. 앞 단계 결과가 다음 단계의 동작 방식을 바꾸는 경우는 단순 파이프라인으로 표현하기 어렵다.

03
역방향 설계
원하는 출력 구조에서 거꾸로 입력과 변환을 설계한다
How — 설계 접근법 Where — 전 레이어
언제 쓰는가
요구사항이 "이런 화면을 보여줘" 또는 "이런 형태의 데이터가 필요해"처럼 출력 중심으로 주어질 때. 데이터 구조가 복잡해서 어디서 시작해야 할지 막막할 때. 개발을 시작했는데 중간에 "이 구조로는 원하는 결과를 낼 수 없다"는 것을 뒤늦게 발견할 때. 역방향 설계는 이 모든 상황에서 시작점을 명확히 한다.
❌ 순방향 (흔한 실수)
원본 데이터 구조 파악
↓
처리 방법 고민
↓
코딩 시작
↓
"이 구조론 안 되네" 발견 (뒤늦게)
VS
✅ 역방향 설계
① 원하는 출력 구조 먼저 정의
↑
② 출력을 만들기 위해 필요한 중간 구조 역추적
↑
③ 중간 구조를 만들기 위해 필요한 원본 확인
↑
④ 입력·변환·출력 구조 확정 후 구현
❌ 순방향 — 원본부터 시작
입력에서 출발해 출력에 도달
# 원본: 주문 목록
orders = [
  {'order_id': 1, 'amount': 50000,
   'customer': 'A', 'date': '2024-01'},
  ...
]

# "뭘 만들어야 하지?"를 생각하며
# 코딩 시작
result = {}
for o in orders:
    # 여기서 막힘
    # 출력 구조가 없으니
    # 어디에 어떻게 담아야 할지 불명확
    pass

# 결국 리팩터링 반복
✅ 역방향 — 출력 먼저 정의
출력 구조 먼저, 역추적으로 설계
# Step 1: 원하는 출력 먼저 정의
# {month: {customer: total_amount}}
wanted_output = {
  "2024-01": {"A": 120000, "B": 80000},
  "2024-02": {"A": 95000},
}

# Step 2: 이 구조를 만들려면
#   month × customer 기준 GROUP BY 필요
#   → SQL: GROUP BY month, customer
#   → Python: defaultdict(lambda: defaultdict(int))

# Step 3: 원본에 month, customer,
#   amount 컬럼 있는지 확인 → OK

# Step 4: 구현 — 명확한 목표로 진행
from collections import defaultdict
result = defaultdict(
    lambda: defaultdict(int)
)
for o in orders:
    result[o['date']][o['customer']] \
        += o['amount']
실전 적용 — SQL 설계에서 특히 강력하다
쿼리를 짤 때 FROM부터 시작하는 습관이 있다면, 역방향 설계를 적용해보라. SELECT 절을 먼저 완성하고 ("어떤 컬럼이 필요한가"), 그 다음 JOIN ("어떤 테이블을 연결해야 그 컬럼을 얻을 수 있나"), 마지막으로 WHERE / GROUP BY를 결정한다. 필요 없는 JOIN과 불필요한 서브쿼리가 자연스럽게 제거된다.
쓰면 안 되는 경우
출력 구조가 아직 확정되지 않았을 때 — 탐색적 분석(EDA) 단계에서는 출력이 유동적이므로 역방향 설계가 오히려 제약이 된다. 또한 실시간 스트리밍처럼 출력이 연속적으로 생성되는 경우, "최종 출력"을 미리 정의하기 어렵다. 역방향 설계는 명확한 출력 명세가 있을 때 빛을 발한다.

04
멱등성 설계
같은 작업을 몇 번 실행해도 결과가 동일하도록 로직을 설계한다
How — 안전성 Timing — 재실행 보장
언제 쓰는가
배치 잡, 데이터 동기화, API 재시도, ETL 파이프라인처럼 같은 작업이 실수로 또는 장애 복구로 두 번 이상 실행될 수 있는 모든 곳. "이 배치가 두 번 돌면 어떻게 되나?"라는 질문에 "두 번 돌면 안 된다"가 아닌 "두 번 돌아도 결과는 같다"고 답할 수 있어야 한다.
멱등성 있음 vs. 없음 — 동일 작업 3회 실행 시
✅ 멱등성 있음
1회 실행
→
잔액: 100,000원
2회 실행
→
잔액: 100,000원 (동일)
3회 실행
→
잔액: 100,000원 (동일)
예: "잔액을 100,000원으로 설정" (SET)
❌ 멱등성 없음
1회 실행
→
잔액: 100,000원
2회 실행
→
잔액: 200,000원 (중복!)
3회 실행
→
잔액: 300,000원 (오염!)
예: "잔액에 100,000원을 추가" (ADD)
❌ 적용 전 — 중복 실행 시 데이터 오염
INSERT — 멱등성 없음
-- 배치가 두 번 실행되면
-- 동일 데이터가 두 번 INSERT됨
INSERT INTO monthly_bonus
    (employee_id, amount, month)
VALUES
    (1001, 500000, '2024-01'),
    (1002, 300000, '2024-01');

-- 1월 배치가 두 번 돌면?
-- 직원 1001이 100만원을 받음
-- 이것을 되돌리려면?
-- "몇 번 실행됐는지"를 추적해야 함
✅ 적용 후 — 몇 번 실행해도 안전
UPSERT — 멱등성 보장
-- (employee_id, month)가 PK
-- 이미 있으면 UPDATE, 없으면 INSERT
INSERT INTO monthly_bonus
    (employee_id, amount, month)
VALUES
    (1001, 500000, '2024-01'),
    (1002, 300000, '2024-01')
ON CONFLICT (employee_id, month)
DO UPDATE SET
    amount = EXCLUDED.amount;

-- 10번 실행해도 결과 동일
-- 장애 후 재실행해도 안전
-- "멱등 키"(employee_id + month)가
-- 중복을 구조적으로 차단
기법 방법 적용 상황
UPSERT 고유 키 기준으로 INSERT OR UPDATE DB 데이터 동기화, 배치 집계 저장
DELETE + INSERT 먼저 기존 데이터를 삭제하고 재삽입 월별·일별 집계처럼 전체를 교체해도 되는 경우
처리 상태 추적 처리 완료 여부를 별도 테이블에 기록, 이미 처리된 건은 스킵 이메일 발송, 결제 요청처럼 외부 사이드이펙트가 있는 경우
멱등 키(Idempotency Key) 요청마다 고유 키를 부여, 동일 키는 한 번만 처리 API 재시도, 결제 API, 외부 서비스 호출
SET 연산 사용 누적(+=) 대신 절대값 설정(=) 방식으로 계산 잔고 업데이트, 재고 수량 반영
멱등성이 어려운 경우 — 외부 사이드이펙트
이메일 발송, SMS, 결제 요청처럼 외부 시스템에 영향을 주는 작업은 단순 UPSERT로 멱등성을 보장할 수 없다. 이 경우 처리 상태 테이블을 두고 "이미 발송됨" 여부를 확인한 뒤 스킵하거나, 외부 API의 멱등 키(Idempotency Key) 기능을 활용한다. 멱등성을 완전히 보장할 수 없다면, 최소한 중복 실행을 감지하고 알릴 수 있는 모니터링을 갖춰야 한다.

네 가지 패턴의 공통점은 규칙·로직·데이터·상태를 분리한다는 것이다. Lookup Table은 규칙을 데이터로, 파이프라인은 로직을 단계로, 역방향은 출력에서 입력으로, 멱등성은 상태를 안전하게 관리한다. 패턴은 외워야 하는 공식이 아니라, 문제를 인식하는 즉시 꺼낼 수 있는 사고의 도구다.
